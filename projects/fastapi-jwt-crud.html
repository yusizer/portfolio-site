FastAPI JWT CRUD API

A production-style REST API: JWT authentication (access + refresh tokens) and full posts CRUD with owner-only permissions — tested end-to-end, with auto-generated Swagger UI.

Stack: FastAPI · PyJWT · bcrypt · SQLAlchemy 2 (async) · Pydantic v2 Delivery: 1–2 weeks Tests: 19 passing

01Context

Almost every product backend needs the same foundation: let users register and log in, issue tokens, and let them create/read/update/delete their own resources safely. This project is that foundation, built the way a real product team would — not a tutorial.

02Problem

Rolling your own auth is where juniors get hurt: plaintext passwords, tokens that never expire, no validation, and "it works on my machine" with zero tests. The job here was to build it so it could actually go to production — secure by default, validated, and provably correct via tests.

03Solution

  • JWT auth: short-lived access tokens + long-lived refresh tokens (HS256), bcrypt-hashed passwords, /auth/register, /auth/login, /auth/refresh, /auth/me.
  • Posts CRUD: create / list / get / update / delete. Update and delete are owner-only — a different logged-in user gets 403, not your data.
  • Validation: Pydantic v2 — email format, min password length, field constraints — bad input returns 422 automatically.
  • DB: SQLAlchemy 2 async — SQLite for dev, PostgreSQL for production (one config switch via DATABASE_URL).
  • Tests: 19 end-to-end cases with pytest + httpx — auth flows, CRUD, owner permissions, duplicate/short-password rejection, invalid tokens.
  • Swagger UI auto-generated at /docs — the API documents itself.

04Outcome

A secure, validated, tested REST API that documents itself — the kind of foundation a client can build a real product on without rewriting auth.

05Tradeoffs & what I'd improve

  • HS256 over RS256 — simpler single-service setup; for multi-service I'd switch to RS256 (asymmetric) so other services can verify tokens without the secret.
  • SQLite for dev — perfect for local/tests; production uses PostgreSQL (one line).
  • Add rate limiting & refresh-token rotation — next hardening step for internet-facing deploys.
