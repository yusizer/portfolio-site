Multi-tenant Telegram CRM — Managed Bots + Rich Messages

A platform bot that provisions a separate Telegram bot per business, each running lead-capture and FAQ flows with AI replies streamed as Rich Messages — all behind a multi-tenant admin dashboard with isolated data and audit logs.

01Context

In April 2026 Telegram shipped Managed Bots (Bot API 9.6) — one bot can create and control other bots — and in June 2026 Rich Messages (Bot API 10.1) for streaming structured AI replies. Almost no portfolio had them yet. The opportunity: build the thing these features exist for — a true multi-tenant CRM where every business gets its own bot, without N separate deployments.

02Problem

Running one Telegram bot per client traditionally means one token, one process, one database per customer — an operational nightmare that doesn't scale and leaks data between tenants. The job was to build it the way a product team would: one platform, many tenants, each strictly isolated, with the new native multi-tenancy the Bot API just enabled.

03Solution

  • Managed Bots (9.6): a manager bot provisions per-customer child bots via getManagedBotToken, KeyboardButtonRequestManagedBot, and t.me/newbot/{manager}/{suggested} links. On managed_bot_created the platform fetches the child token, encrypts it at rest (Fernet), and spawns a polling task.
  • Rich Messages (10.1): replies stream via sendRichMessageDraft (ephemeral 30s preview) then finalize with sendRichMessage — bold headings, bullet lists, code blocks, built through a poka-yoke HTML builder that auto-escapes all user content.
  • Multi-tenancy, two paths: the real Managed Bots flow and a simulated /demo path (one bot switches tenants via inline keyboard) so it's demoable in a single chat without creating live child bots.
  • FSM flows: lead capture (name → contact → service → note), FAQ (RAG-lite over a per-tenant knowledge base), and an onboarding wizard — built with StatesGroup, not a framework, per the "simple composable patterns over bloat" principle.
  • FastAPI admin: tenants overview, per-tenant leads with status workflow, 7-day analytics, FAQ CRUD. Every query filters by tenant_id — a tenant can never see another's data. Session auth + append-only audit log.
  • Tests: 27 cases — the lead state machine, tenant isolation at the data and HTTP layers, provisioning with encrypted-token verification, the Rich builder, and admin CRUD.

04Outcome

A flagship that ships two Bot API features weeks after their release — native multi-tenancy, streaming rich replies, encrypted secrets, and provable tenant isolation. The kind of build that reads as senior because it leans on the platform's newest primitives instead of reinventing them.

05Tradeoffs & what's next

  • Per-bot long-polling over webhooks — simpler for an MVP; webhooks scale further and are a Phase 3 swap.
  • Demo-mode LLM (no API key) — RAG-lite retrieval with templated rich answers; an OpenAI key is a one-function swap-in at answer_faq to ground real GPT replies.
  • Phase 2: booking FSM, channel auto-posting scheduler, Redis FSM storage, richer analytics. Phase 3: tenant subscription payments (Stripe + Telegram Stars), moderation queue, token rotation.
