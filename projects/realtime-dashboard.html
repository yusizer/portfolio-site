Real-time Live Dashboard

A FastAPI dashboard that streams live metrics to every connected browser over WebSockets — numbers update every second and a sparkline draws itself as data arrives. A REST snapshot endpoint serves non-real-time consumers.

Stack: FastAPI · WebSockets · uvicorn · vanilla JS + canvas Delivery: 1 week Tests: 4 passing

01Context

Ops and product teams want to see what's happening right now — requests per second, active users, error rate — not a stale report from yesterday. The job here was a dashboard that reflects reality in real time, shared across whoever is watching.

02Problem

Polling a REST endpoint every second is wasteful and laggy — many requests, delayed updates, no shared sense of "now". A push-based approach (WebSockets) keeps every viewer in sync with a single stream and minimal overhead, but needs correct connection handling and a clean background task lifecycle.

03Solution

  • WebSocket stream: /ws accepts a connection, sends an immediate snapshot, then pushes updated metrics every second — so the client is never blank.
  • Background broadcaster: a single asyncio task advances the metrics and is started/stopped via FastAPI's lifespan (no leaked tasks on shutdown).
  • REST snapshot: GET /api/metrics returns the latest values for scripts, alerts, or embedding where WebSockets don't fit.
  • Live UI: metric cards + a canvas sparkline that auto-draws as values arrive; auto-reconnects if the socket drops.
  • Tests: REST snapshot, dashboard page, and a WebSocket stream test (Starlette TestClient).

04Outcome

A real-time dashboard where every connected viewer sees the same live numbers ticking together — push-based, low-overhead, with a REST fallback for non-real-time consumers.

05Tradeoffs & what I'd improve

  • Simulated metrics — swap _tick() for real reads (DB counters, Redis, another API); the WebSocket plumbing is unchanged.
  • Single-process broadcast — fine for one server; for multiple workers behind a load balancer, move pub/sub to Redis so all workers share one stream.
  • Add alerting thresholds — flash a card red when error rate exceeds a limit.
